# Apex telemetry CSV interchange format

The **Apex telemetry CSV** is the single on-disk interchange format for
*measured* car telemetry (laps imported from real data sources such as FastF1)
and for the standard-format output of the correlation tools. It is defined once
here and consumed by `apex-correlate` (import/resample/write) and produced by
`tools/fastf1_export.py`.

It is deliberately a close cousin of the simulation CSV written by
`apex-telemetry` (`export_qss_csv` / `export_columns_csv`): both use `#` comment
headers, both name columns by **channel-registry names**, and both carry a
`# columns:` units line. The difference is provenance (see
[Measured data vs. sim artifacts](#measured-data-vs-sim-artifacts)).

## File shape

```
# <key>: <value>          ← zero or more free-form provenance comment lines
# ...
# grid: s                 ← REQUIRED: the time base, `s` or `t`
# columns: s[m], speed[m/s], throttle[], ...   ← REQUIRED units line
#                         ← one blank comment line (separator)
s,speed,throttle,...      ← header row: registry channel names
0.0,72.3,1.0
2.5,74.1,1.0
...
```

A comment-aware reader (`csv::ReaderBuilder::comment(Some(b'#'))`) skips every
`#` line transparently and lands on the header row.

### 1. Comment header (`# key: value`)

Zero or more lines of the form `# <key>: <value>`. These are free-form and are
carried through import→write unchanged (order preserved). Two keys are
**reserved** and given dedicated meaning:

- `# grid: s` or `# grid: t` — **required**, see below.
- `# columns: ...` — **required**, the units line, see below.

Any other key (`year`, `gp`, `session`, `driver`, `lap`, `fastf1_version`,
`source`, …) is preserved verbatim as provenance. This block is
format-compatible with `apex-telemetry`'s provenance block, so a simulation
CSV's `# config_hash: …`, `# git_sha: …`, etc. lines are also valid header
lines here — but see the provenance distinction below: **measured** telemetry
files must NOT carry sim `*_hash` provenance.

### 2. Grid declaration (`# grid:`)

```
# grid: s      # rows are uniform in arc-length s (metres)
# grid: t      # rows are uniform in time t (seconds)
```

**Required.** Declares which channel is the monotonic sample axis and in what
coordinate the rows are (intended to be) uniform:

- `grid: s` → the file has an `s` column; rows step along track distance.
- `grid: t` → the file has a `t` column; rows step along elapsed time.

A file may contain *both* `s` and `t` columns (FastF1 gives us both); `# grid:`
only names which one is the resampling/interpolation axis. Import falls back to
inferring the grid from the present axis channel only if `# grid:` is absent, and
errors if neither `s` nor `t` is present.

### 3. Units line (`# columns:`)

```
# columns: s[m], speed[m/s], throttle[], brake[], gear[], rpm[rpm]
```

Generated by `apex_telemetry::channels::csv_columns_comment`. One `name[symbol]`
pair per data column, in header order, where `symbol` is the **registry unit
symbol** for that channel (empty brackets `[]` for dimensionless channels like
`throttle`). This line documents the canonical unit each column is stored in; a
column name not in the registry renders `name[?]`.

### 4. Header row + data rows

- **Header row**: comma-separated **registry channel names** (e.g.
  `s,speed,throttle`). Every column name MUST be a
  `apex_telemetry::ChannelId::name()`; unknown names are rejected (or ignored)
  per the importer's `unknown_columns` policy.
- **Data rows**: comma-separated `f64` values, one row per sample. All columns
  have the same number of rows.

## Canonical units (the registry contract)

Column values are stored in the **registry unit** for that channel — the unit
named in the `# columns:` line. A producer that measures in another unit
converts *before* writing (e.g. `tools/fastf1_export.py` converts km/h→m/s and
percent→fraction so the CSV is registry-canonical). The generic importer
(`import_telemetry`) can instead convert on the way in, driven by a
channel-mapping config that declares each source column's unit; see
`apex-correlate` and the mapping schema. Supported conversions include
km/h→m/s, deg→rad, percent→0–1 fraction, ms→s, and identity, all routed through
`Unit::si_factor()`.

## Lap boundaries

**Single lap per file, for now.** Each CSV holds exactly one lap. Multi-lap
files and lap slicing arrive with the track-alignment work; until then, a lap is
a whole file and `s`/`t` start at (approximately) zero.

## Missing / non-finite samples

Measured telemetry has real gaps (dropouts, sensor faults). Non-finite values
(`NaN`) in **measured** columns are **kept as `NaN`** on import rather than being
dropped or interpolated away — a gap is data, not an error. Consumers inspect a
per-channel validity mask / NaN count (`Telemetry::validity_mask`,
`Telemetry::nan_count`). Resampling does **not** interpolate across a gap longer
than a configurable threshold; it emits `NaN` there instead of inventing values.

## Measured data vs. sim artifacts

`apex-telemetry`'s simulation CSVs carry a **`RunMetadata` provenance block** —
`config_hash`, `car_hash`, `track_hash`, `settings_hash`, `git_sha`,
`apex_version`, `seed`, `timestamp` — because they are reproducible artifacts of
a specific sim configuration. **Measured** telemetry is not a sim artifact:
there is no car/track/settings hash to compute, and fabricating one would be
misleading. Measured files therefore carry only **descriptive** provenance
(data source identifiers: year, GP, session, driver, lap, and the exporter
version) and MUST NOT include the sim `*_hash` fields. The `git_sha` /
`timestamp` of the exporting tool MAY be recorded as ordinary comment lines but
are not part of a `RunMetadata` block.

### Hybrid provenance on correlation artifacts

Some artifacts mix the two. The `apex-14 correlate` overlays (speed-vs-`s` SVG,
driver-inputs SVG) compare a **sim** trace against a **measured** trace, so they
carry **both**:

- a full sim-side `RunMetadata` block (as an SVG `<metadata>` element) —
  `config_hash` / `car_hash` / `track_hash` / `settings_hash` etc. — describing
  the QSS run, which *is* reproducible-by-hash; plus
- **descriptive** measured-source lines (session identifiers from the measured
  file header) as an XML comment — never sim hashes for the measured side.

This hybrid is deliberate: the sim inputs are hash-pinned, the measured inputs
are descriptive. The generated `report.md` mirrors it — a `RunMetadata` hash
block for the sim and a plain key/value list for the measured source.

### Inferred CSVs (`apex-14 infer`)

The QSS channel-inference output is **derived-from-measured**: it carries the
measured source header verbatim plus **descriptive** inference provenance — no
`RunMetadata` sim block. Its `# key: value` header adds:

- `inference: apex-14 infer v<X>` and `infer_driven_line: direct|offset`;
- `infer_car_file:` and `infer_car_hash:` — the fitted car's file and
  `car_params_hash` (a single descriptive hash line, not a full `RunMetadata`);
- `inference_caveat:` — the **mandatory** effective-parameter caveat: the
  inferred aero/downforce/loads/power derive from the fitted *effective*
  coefficients and are model-consistent estimates, **not measurements**.

The data columns are the (re-gridded) input channels plus the inferred channels
(`downforce`, `aero_drag_force`, `fz_front`, `fz_rear`, `grip_util`,
`tractive_power`, `braking_power`, and `lateral_g`/`longitudinal_g`).
